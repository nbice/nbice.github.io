Understanding ActiveRecord Callbacks: after_commit vs after_create

Published: March 15, 2024

When working with ActiveRecord callbacks in Rails, understanding the difference between `after_commit` and `after_create` is crucial for avoiding subtle bugs in your application.

The key distinction lies in transaction handling:

  • `after_create` runs within the same transaction as the record creation
  • `after_commit` runs only after the transaction has been committed to the database

This becomes particularly important when dealing with background jobs or external services. Using `after_create` might cause issues if the transaction rolls back, while `after_commit` ensures the record exists in the database before any subsequent operations.

class User < ApplicationRecord
  # Might cause issues if transaction rolls back
  after_create :send_welcome_email

  # Safer approach - ensures user exists in DB
  after_commit :send_welcome_email, on: :create
end
